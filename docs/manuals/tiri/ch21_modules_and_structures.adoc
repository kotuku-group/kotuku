[[ch-modules-and-structures]]
== Modules & Structures
indexterm:[modules] indexterm:[structures] Tiri provides direct access to Kōtuku's compiled modules through the `mod` namespace.  This chapter defines the interfaces for loading modules, calling module functions, working with C/C++ structures, and interpreting error codes.

The Core module (`mSys`) is loaded automatically and is always available.  All other modules are loaded on demand with `mod.load()` or `mod.new()`.  The `include` directive (<<sec-the-include-directive>>) provides a lightweight alternative that loads only definitions (constants, structures, and class metadata) without opening the module's function table.

[[sec-loading-modules]]
=== Loading Modules
==== mod.load()

[.api]`module = mod.load(Name)`

indexterm:[modules, loading] Loads the Kōtuku module identified by `Name` and returns a module interface object.  `Name` is a case-insensitive alphanumeric string with a maximum length of 32 characters.  Passing an invalid name raises an exception with `ERR_Syntax`.

If the module cannot be loaded (e.g. the binary is missing or initialisation fails), an exception is raised with `ERR_LoadModule`.

Loading a module automatically processes its IDL definitions, making all constants and structure definitions available.  A separate `include` call for the same module is unnecessary.

[source,tiri]
----
mGfx = mod.load('display')
err, x, y = mGfx.getCursorPos()
----

[discrete]
==== Module Protection

Each module is loaded at most once per process.  Subsequent calls to `mod.load()` with the same name return the existing module interface without reloading.  Use the `?=` operator to guard against redundant calls:

[source,tiri]
----
mGfx ?= mod.load('display')
----

[discrete]
==== Naming Conventions

Module interfaces should always be stored in consistently named variables at the top-level scope.  The following table lists the standard naming conventions for the default set of modules:

[cols="1,1a",options="header"]
|===
|Module |Global Name
|audio
|`mAudio`
|core
|`mSys` 💡
|display
|`mGfx`
|font
|`mFont`
|network
|`mNet`
|vector
|`mVec`
|xml
|`mXML`
|===

💡The Core module (`mSys`) is an essential API that is always loaded by default.

[discrete]
==== Calling Module Functions

Module functions are accessed through dot notation on the module interface.  Function lookup is case-insensitive.

[source,tiri]
----
mGfx = mod.load('display')
err, x, y = mGfx.getCursorPos()
----

Most module functions return an `ERR` code as their first result, followed by any additional return values.  See <<sec-error-codes>> for the full error code reference and <<ch-error-handling>> for general advice.

Error codes at or above `ERR_ExceptionThreshold` are automatically promoted to exceptions when the call is enclosed in a `try` statement.

[discrete]
==== Multiple Return Values

Some module functions return multiple results.  In C/C++, result parameters are written to pointers.  In Tiri, these pointer parameters are removed from the call signature and their values are appended to the result set:

[source,tiri]
----
-- C prototype: ERR ListMemory(INT Flags, ListMemory **Memory)
-- Tiri call:
err, list = mSys.ListMemory(0)
----

Functions that return allocated resources are automatically covered by the garbage collector.  Use `local` scoping to ensure timely cleanup.

[discrete]
==== Buffer Handling

When calling functions that copy results to a user-supplied buffer, pass an array or allocated string type.  Tiri automatically provides the buffer size to the function if the corresponding size parameter is omitted:

[source,tiri]
----
buffer = array<byte, file.size>
err, result = file.acRead(buffer)
print(tostring(buffer))

buffer = string.alloc(file.size)
err, result = file.acRead(buffer)
print(buffer)
----

[[sec-the-include-directive]]
=== The include Directive
[.api]`include 'ModuleName'[, 'ModuleName' pass:[...]]`

indexterm:[include directive] The `include` directive loads the interface definitions (constants, structures, and class definitions) for one or more Kōtuku modules at compile time.  The parser processes each named module while parsing the source and emits no runtime bytecode for the statement.

`include` accepts one or more module names as a comma-separated list of string literals:

[source,tiri]
----
include 'xml', 'display'
----

Each module name must be a non-empty alphanumeric string.  Variables, expressions, or invalid names raise a parse error.

[discrete]
==== Semantics

* Definitions are loaded from the module's IDL (Interface Definition Language) registry.
* The statement is processed at parse time and has no runtime result.
* `include` may appear wherever a statement is valid.  Inside a function body, it still loads definitions during parsing rather than when the function is called.
* An `include` list loads each module named in the list from left to right.
* Each module is loaded at most once per session.  Repeated `include` calls for the same module are silently ignored.
* Calling `mod.load()` automatically loads the corresponding module definitions, making an explicit `include` unnecessary in many cases.

[discrete]
==== When to Use include

Use `include` when access to a module's constants, structures, or class definitions is needed without loading the module's function table.  If the function table is also required, prefer `mod.load()` (see <<ch-modules-and-structures>>), which loads definitions implicitly.

[source,tiri]
----
-- Access display constants without loading the function table
include 'display'
flags = DPMS_OFF -- The DPMS_OFF constant originates from the display module

-- Equivalent: mod.load() loads definitions and also returns the interface
mGfx = mod.load('display')
----

[[sec-type-conversion]]
=== Type Conversion
indexterm:[type conversion] When calling a module function, Tiri performs automatic type conversion to match the types declared in the function definition.  For example, a string value passed to an integer parameter is converted to an integer.  A boolean passed to a string parameter is converted to its string representation.

If the conversion is not possible (e.g. an abstract pointer cannot be converted to a number), an exception is raised with `ERR_WrongType`.

While automatic conversion is convenient, prefer passing correctly typed values for efficiency.  Automatic conversion imposes a small overhead on every mismatched call, and may not be supported by the JIT optimiser.

[cols="1,1,2a",options="header"]
|===
|From |To |Behaviour

|string
|integer / double
|Parsed as a numeric literal.  Raises an exception if the string is not a valid number.

|number / boolean
|string
|Converted to its string representation.

|object
|integer
|The object's unique ID is used.
|===

[[sec-structures]]
=== Structures
indexterm:[structures] The structure interface allows Tiri scripts to create and manipulate C/C++ structures declared in the Kōtuku API.  Many class and module functions use structures for returning condensed information.  All structure functions are accessed through the `struct` namespace.

A structure instance is a first-class value with its own type.  The `type()` function returns `"struct"` for a structure instance (see <<sec-structs-type>>).  A structure is a reference type: assigning it to another variable copies the reference, not the payload.  Field access is provided directly on the instance; a structure is not a table and does not respond to table operations other than those defined in this section.

[cols="1,3a",options="header"]
|===
|Property |Guarantee

|Mutation
|Field writes modify the backing memory in place.  Reads and writes to a field are visible through every reference to the same instance.

|Ordering
|Fields have a fixed order fixed by the structure definition.  `pairs()` visits fields in declaration order.

|Iteration
|`pairs()` yields each field name and its current value.  The set of fields is fixed by the definition and does not change during iteration.

|Identity
|Two references are equal via `is` when they refer to the same instance.  Distinct instances are never equal, even when their payloads are identical.

|Equality
|Structures are compared by reference identity.  Field-by-field comparison is not performed.

|Performance
|Field access by name is O(1).  Reads and writes of scalar fields on a value annotated as `struct` compile to dedicated bytecodes (see <<ch-performance-and-the-jit>>).
|===

==== struct.new()

[.api]`instance = struct.new(StructName, [Fields])` +
[.api]`instance = struct.new(StructName, Address)`

Creates a new structure instance from a registered definition identified by `StructName`.  The definition must have been registered through an `include` directive, a `mod.load()` call, or a prior call to `MAKESTRUCT()`.  Raises an exception if the definition is not found.  Returns a value whose `type()` is `"struct"`.

If `Fields` is a table, the key-value pairs are applied to the corresponding structure fields during creation.  Field values are subject to the assignment rules defined in <<sec-struct-field-access>>; a value that the target field does not accept raises an exception.

If `Address` is provided as a pointer value, the structure references existing memory at that address instead of allocating new memory.

[source,tiri]
----
include 'vector'

-- Create with default (zeroed) values
stop = struct.new('GradientStop')
stop.offset    = 0.5
stop.rgb.red   = 1.0
stop.rgb.green = 0.0
stop.rgb.blue  = 0.0
stop.rgb.alpha = 1.0

-- Create with initial field values
stop = struct.new('GradientStop', { offset = 0.5 })

-- Create from an existing memory address
tag = struct.new('XMLTag', address)
----

==== struct.size()

[.api]`bytes = struct.size(StructName)`

Returns the byte size of the structure definition identified by `StructName`.  Does not create a structure instance.  Raises an exception if the definition is not found.

==== struct.clone()

[.api]`copy = struct.clone(Source)`

Creates a new structure instance of the same definition as `Source` and deep-copies its payload into the new instance.  Owned C++ string and vector fields are duplicated so that the copy holds independent storage; subsequent modifications to `Source` do not affect the returned instance.  Raises an exception if `Source` holds an unsupported C++ array field type.

[source,tiri]
----
source = struct.new('CopyRecord', { count = 12, name = 'source' })
copy   = struct.clone(source)
source.name = 'changed'
print(copy.name)   -- 'source'
----

==== struct.copy()

[.api]`dest = struct.copy(Dest, Source)`

Deep-copies the payload of `Source` into the existing `Dest` instance and returns `Dest`.  `Dest` and `Source` must share the same definition; a mismatch raises an exception with `ERR_Mismatch`.  Raises an exception if either payload is null, if the payloads overlap in memory, or if the definition holds an unsupported C++ array field type.

==== structSize() Method

[.api]`bytes = inst.structSize()`

Returns the byte size of the structure instance.  Be wary of confusing this method with `size()`, which does not exist in this context (`size` is a common field name).

==== ptr() Method

[.api]`address = inst.ptr()` +
[.api]`old_address = inst.ptr(NewAddress)`

Retrieves or replaces the memory address backing the structure.  Returns `nil` if the current address is zero.

Changing the address avoids allocating a new structure instance for each pointer to be processed.  This is useful when iterating over arrays of structures in memory:

[source,tiri]
----
tag = struct.new('XMLTag', first_address)
-- Process tag...
tag.ptr(next_address)
-- Process next tag using the same instance...
----

WARNING: Modifying the pointer referenced by the struct is dangerous.  Alternative usage patterns are usually available so that this anti-pattern can be avoided.

==== Length Operator

The `#` operator returns the total number of fields in the structure.

[[sec-struct-field-access]]
==== Field Access

Fields are read and written directly on the instance using dot notation.  Field names are matched against the definition in camel-case.  Reading or writing a name that the definition does not declare raises an exception.

The value produced by a read, and the values accepted by a write, depend on the field's declared type:

[cols="1,3a",options="header"]
|===
|Field category |Behaviour

|Scalar numeric (`l`, `d`, `x`, `f`, `w`, `b` and unsigned variants)
|Reads and writes a number.  Unsigned fields preserve the full unsigned range of the storage width.

|Object (`o`)
|Reads as a native object (`type()` is `"object"`), or `nil` when unset.  Accepts an object or `nil` on assignment; assigning `nil` clears the field.  Assigning a detached object wrapper raises an exception.

|Owned C++ string (`zs`)
|Reads and writes a string.  The structure owns the string storage.

|Raw C string (`s`)
|Reads a string.  Assignment is not supported and raises an exception.

|Char array (`c[N]`)
|Reads as a string.  Accepts a string or byte array on assignment.

|Pointer (`p`) and function reference (`r`)
|A null pointer reads as `nil`; a non-null pointer or function reference reads as userdata.  A function reference (`r`) does not accept assignment and raises an exception.

|Embedded structure (`e`)
|Reads as a structure view over the embedded field, or an array of views for an embedded array.  The view keeps the enclosing instance alive.
|===

Array fields accept an array value on assignment:

* A fixed array (`type[N]`) copies elements from the source array.  A source shorter than `N` clears the remaining elements; an empty array clears the field.  A source longer than `N` raises an exception.
* A C++ vector field (`z`-prefixed array, e.g. `zl`, `zs`) resizes to match the source array and copies its elements.
* An element type that does not match the field's declared type raises an exception.

[source,tiri]
----
value.fixed = array<int> { 4, 5, 6 }   -- copies three elements
value.fixed = array<int> { 9 }         -- { 9, 0, 0 }
value.dynamic = array<int> { 8, 9, 10 } -- C++ vector resizes and copies
----

[[sec-struct-iteration]]
==== Iteration

`pairs()` iterates the fields of a structure in declaration order, yielding each field name and its current value.

[source,tiri]
----
value = struct.new('IterableRecord', { count = 7, name = 'sample' })
for name, field_value in pairs(value) do
   print(name, field_value)
end
----

[[sec-struct-tostring]]
==== String Representation

`tostring()` produces a string in the form `StructName: 0xADDRESS`, where `StructName` is the definition name and `ADDRESS` is the payload address in hexadecimal.

[source,tiri]
----
tostring(struct.new('IterableRecord'))  -- 'IterableRecord: 0x...'
----

==== Custom Structures

===== MAKESTRUCT()

[.api]`MAKESTRUCT(Name, Definition)` 🧪

Registers a custom structure definition.  `Name` is the identifier used in subsequent `struct.new()` and `struct.size()` calls.  `Definition` is a comma-separated string of field specifications in declaration order.

Each field specification consists of an optional unsigned prefix (`u`), a type character, and the field name in camel-case.  Pointer fields may include a `:StructName` suffix to reference another registered structure.

[source,tiri]
----
MAKESTRUCT('XMLTag', 'lIndex,lID,pChild:XMLTag,pPrev:XMLTag,pNext:XMLTag,pPrivate,pAttrib,wTotalAttrib,uwNest')

tag = struct.new('XMLTag')
tag.index = 42
----

===== Variant Prefixes

[cols="1,3a",options="header"]
|===
|Prefix |Description

|`u`
|Unsigned variant of the following type, e.g. `uw` for unsigned 16-bit word.

|`z`
|C++ variant, e.g. `zs` for `std::string`.
|===

===== Fixed Arrays

Append `[N]` to a field name to declare a fixed-size array of `N` elements.  Use `[0]` for a pointer to a null-terminated array:

[source,tiri]
----
MAKESTRUCT('Buffer', 'bData[256]')
MAKESTRUCT('StringList', 'pStrings[0]')
----
